\begin{abstract}
This document describes a layered experiment plan for replacing NVIDIA cuVS/cuRAFT with
AMD hipVS/hipRAFT in the Milvus/Knowhere GPU path. It complements the CPU benchmarking
runbook (\texttt{ann\_framework\_runbook.tex}): FAISS and Milvus Standalone CPU results on
\texttt{amd-rx7900xtx} are the control; GPU port work proceeds in five layers from ROCm sanity
through standalone libraries, Knowhere, Milvus source builds, and harness benchmarks.
Layer~1 completed on the 7900 XTX lab host (ROCm~7.0.2, July~2026) with documented RDNA3 gtest
caveats.
\end{abstract}

\section{Background}

Milvus does not call cuVS directly from Python. The stack is:

\begin{verbatim}
pymilvus / VectorDBBench
        |
   Milvus server
        |
   Knowhere  (vector engine: CPU + GPU indexes)
        |
   GPU path: GPU_IVF_FLAT, GPU_IVF_PQ, GPU_CAGRA, GPU_BRUTE_FORCE
        |
   NVIDIA: RAFT / cuVS
   AMD goal: hipRAFT + hipVS (ROCm-DS ports)
\end{verbatim}

\textbf{hipVS} is an AMD port of cuVS
(\url{https://github.com/ROCm-DS/hipVS},
\url{https://rocm.docs.amd.com/projects/hipVS/en/latest/}) with API compatibility as an
explicit goal. \textbf{hipRAFT} is the matching primitives layer (RAFT/cuRAFT analogue on
ROCm).

The stock \textbf{Milvus v2.5.4 CPU Docker image} used for CPU baselines does \emph{not}
exercise the GPU/HIP path. GPU experiments require \textbf{source builds} (or a custom GPU
image you build yourself). NVIDIA Milvus GPU images will not run on AMD.

\section{Prerequisites (CPU control)}

Before starting GPU layers, the following should be in place on the target AMD host
(see the main runbook for commands and measured tables):

\begin{itemize}[leftmargin=*]
  \item SIFT-1M HDF5 dataset and in-repo FAISS harness (\texttt{run\_faiss\_hdf5.py}).
  \item Milvus Standalone Docker CPU (\texttt{milvus-docker/}, \texttt{user.yaml}
        \texttt{knowhere.simdType: sse4\_2}).
  \item Optional but recommended: VectorDBBench \texttt{milvusivfflat} IVF \texttt{nprobe}
        sweep for same-host FAISS vs Milvus CPU comparison.
\end{itemize}

On \texttt{amd-rx7900xtx} (7900 XTX / \texttt{gfx1100}), CPU baselines and the IVF sweep
were captured 2026-06-30. GPU work does not replace those numbers; it adds a new backend
to compare against them.

\section{Layered strategy}

Do not attempt to ``replace cuVS in Milvus'' in one step. Use \textbf{five layers}
(Layer~0 through Layer~4), each with explicit pass/fail criteria. Only advance when the
current layer passes.

\subsection{Upgrade to ROCm 7.0.2 (required for Layer~1 on gfx1100)}
\label{sec:layer1-rocm7-upgrade}

AMD does \textbf{not} support in-place ROCm upgrades. On Ubuntu~22.04 for Radeon + ROCm:

\begin{enumerate}[leftmargin=*]
  \item Install \texttt{dkms} and kernel headers if missing:
        \texttt{sudo apt install dkms linux-headers-\$(uname -r)}.
  \item Uninstall old stack: \texttt{sudo amdgpu-uninstall} (or \texttt{apt autoremove rocm*}).
  \item Remove stale repo files under \texttt{/etc/apt/sources.list.d/}; \texttt{apt clean}.
  \item Install \texttt{amdgpu-install} 7.0.2 \texttt{.deb} from AMD; run

  \item Reboot; add user to \texttt{render,video}; verify \texttt{rocminfo}, \texttt{hipcc},
        and \texttt{/opt/rocm/include/hipcub/device/device\_for.hpp} exists.
\end{enumerate}

Full command transcript: AMD Radeon install guide for ROCm~7.0.2 on Jammy
(\url{https://rocm.docs.amd.com/projects/radeon-ryzen/en/docs-7.0.2/docs/install/installrad/native_linux/install-radeon.html}).

\section{Layer 1 --- hipRAFT and hipVS in isolation}
\label{sec:layer1-hipvs-hipraft}

\subsection{Goal}

Prove AMD libraries work on the GPU \textbf{before} Knowhere or Milvus. Layer~1 success means:
\texttt{libcuvs.so} builds for \texttt{gfx1100}, hipVS tests run on GPU, and IVF/brute-force search
returns sensible neighbors.

\subsection{Reality check (7900 XTX / RDNA3)}

\begin{itemize}[leftmargin=*]
  \item \textbf{ROCm~$\geq$7.0.2 is required} for \texttt{release/rocmds-25.10} on this branch.
        ROCm~6.4.4 hipCUB lacks \texttt{cub::DeviceFor} (\texttt{device\_for.hpp}); hipVS
        configure/compile fails with \texttt{no member named 'DeviceFor' in namespace 'hipcub'}.
  \item \textbf{RDNA3 / \texttt{gfx1100} requires warp size 32.} hipCollections (cuco) defaults to
        CDNA-style warp size 64; without \texttt{USE\_WARPSIZE\_32=ON} configure fails with
        \texttt{not supported with warp size 64 (gfx1100)}. This path is \textbf{experimental}
        in upstream cuco.
  \item Official hipVS docs list \textbf{Instinct GPUs only} (\texttt{gfx90a}/\texttt{gfx942});
        \texttt{gfx1100} is community/experimental. Expect gtest gaps even when search works.
  \item First build downloads dependencies via ROCmDS-cmake and can take \textbf{1--3+ hours} and
        \textbf{10--30\,GB} disk under \texttt{\textasciitilde{}rocmds\_check\_gfx1100/}.
\end{itemize}

\subsection{Step 2b --- Patch rocThrust version for ROCm 6.4.x only}

\textbf{Skip this step on ROCm~7.x} (system rocThrust is \textbf{4.0.0}; leave
\texttt{versions.json} unchanged).

On ROCm~6.4.4 only: hipRaft pins rocThrust \textbf{4.0.0} but apt ships \textbf{3.3.x}.
Without the patch, \texttt{find\_package} fails and CPM tries the sentinel git tag
\texttt{We-always-use-find\_package-for-rocthrust.} (which is not a real branch).

\begin{verbatim}
python3 <<'PY'
import json
from pathlib import Path
p = Path.home() / "rocmds_check_gfx1100/hipRaft/versions.json"
data = json.loads(p.read_text())
data.setdefault("packages", {}).setdefault("rocthrust", {})["version"] = "3.3.0"
p.write_text(json.dumps(data, indent=2) + "\n")
print("patched", p)
PY
\end{verbatim}

\subsection{Step 9 --- Recall sanity vs FAISS (float, small data)}

After Layer~1 builds pass, validate \textbf{float} IVF (what SIFT and \texttt{GPU\_IVF\_FLAT}

then add a small hipVS or \texttt{cuvs} Python script when pursuing Knowhere integration.

\subsection{How to read \texttt{NEIGHBORS\_ANN\_IVF\_FLAT\_TEST} output}
\label{sec:ivf-gtest-decode}

Each gtest case (\texttt{AnnIVFFlat/NN}) runs three sub-tests in one \texttt{TEST\_P}:
\texttt{testIVFFlat()} (search/recall), \texttt{testPacker()} (GPU list packing math),
\texttt{testFilter()}. A single \texttt{[ FAILED ]} line does \textbf{not} mean IVF search is
broken.

\paragraph{Recall line (usually good news).}
\begin{verbatim}
[RAFT] [info] Recall = 1.000000 (10000/10000), ... below the threshold (eps = 0.001000).
\end{verbatim}
This checks \textbf{neighbor correctness}. Values near \textbf{1.0} mean GPU IVF search found
the right neighbors. This is what matters for SIFT / \texttt{GPU\_IVF\_FLAT}. \textbf{Do not}
try to fix failures by changing \texttt{eps}---recall often already passes.

\paragraph{Packer failure at \texttt{ann\_ivf\_flat.cuh:376} (RDNA3 gap).}
\begin{verbatim}
Failure
Value of: thrust::reduce(..., mask, ...) == list_size * ps.dim
  Actual: false
  Expected: true
\end{verbatim}
This is \textbf{not} a recall threshold. It counts \texttt{true} bits in an internal packing
\texttt{mask} and expects \textbf{exactly} \texttt{list\_size $\times$ dim}. On \texttt{gfx1100}
with \texttt{USE\_WARPSIZE\_32=ON}, this assert fails for \textbf{both float and uint8} suites
even when recall is \textbf{1.0}. Upstream tests Instinct (\texttt{gfx90a}/\texttt{gfx942}) only.

\paragraph{\texttt{GetParam()} decode.}
Example: \texttt{\{ 1000, 10000, 16, 10, 512, 1024, 0, 0,0,0 \}} means
\texttt{num\_queries=1000}, \texttt{num\_db\_vecs=10000}, \texttt{dim=16}, \texttt{k=10},
\texttt{nprobe=512}, \texttt{nlist=1024}, \texttt{metric=L2}, plus three booleans. The index
\texttt{/93} is just parameter set \#93, not an error code.

\paragraph{Do not patch gtests to pass.} Lowering recall \texttt{eps} or commenting out the
packer \texttt{ASSERT\_TRUE} hides a real layout bug on extend/packer paths. For Milvus work,
use \textbf{float} vectors and application-level recall tests instead.

\subsection{Layer~1 pass criteria}

On \textbf{Instinct} (\texttt{gfx90a}/\texttt{gfx942}), upstream expects both gtests green. On
\textbf{RDNA3} (\texttt{gfx1100}), use the pragmatic bar:

\begin{itemize}[leftmargin=*]
  \item hipRAFT \texttt{libraft.so} and hipVS \texttt{libcuvs.so} build for \texttt{gfx1100}
        with ROCm~$\geq$7.0.2.
  \item \texttt{BRUTEFORCE\_C\_TEST} exits~0 (required on gfx1100).
  \item \texttt{NEIGHBORS\_ANN\_IVF\_FLAT\_TEST}: recall lines may show 1.0 even when gtest
        fails packer asserts---document, do not patch tests.
  \item \texttt{rocm-smi} shows VRAM activity during tests.
  \item Record branch, ROCm version, build flags, log paths (Table in
        Section~\ref{sec:layer1-measured}).
\end{itemize}

Repositories:
\begin{itemize}[leftmargin=*]
  \item \url{https://github.com/ROCm-DS/hipRaft} (branch \texttt{release/rocmds-25.10})
  \item \url{https://github.com/ROCm-DS/hipVS} (branch \texttt{release/rocmds-25.10})
  \item Build docs: \url{https://rocm.docs.amd.com/projects/hipVS/en/latest/install/build.html}
\end{itemize}

\textbf{Fail fast if:} hipVS lacks \texttt{GPU\_IVF\_FLAT}/brute-force coverage needed for your
first Knowhere target---check upstream before Layer~2 fork work.

\section{Layer 2 --- Knowhere with HIP backend}

\subsection{Goal}

Substitute the library link at the Knowhere layer, still without full Milvus.

\subsection{Steps}

\begin{enumerate}[leftmargin=*]
  \item Clone \textbf{milvus-io/knowhere} pinned to your Milvus release (e.g.\ v2.5.x).
  \item Study CMake options for GPU / cuVS / RAFT in the NVIDIA build docs.
  \item Fork or patch: link \texttt{hipVS} / \texttt{hipRAFT} instead of \texttt{cuvs} /
        \texttt{raft}; add backend flag (e.g.\ \texttt{KNOWHERE\_WITH\_HIP} vs
        \texttt{KNOWHERE\_WITH\_CUVS}).
  \item Run Knowhere unit tests for one index, e.g.\ \texttt{GPU\_IVF\_FLAT}:
        build on GPU, search with \texttt{nprobe} sweep, compare recall curve to CPU
        \texttt{IVF\_FLAT} on the same data.
\end{enumerate}

\textbf{Pass:} Knowhere tests green; recall vs \texttt{nprobe} rises like CPU IVF (not
necessarily bit-identical).

This is where ``replace cuVS/cuRAFT'' actually happens---in Knowhere's build and GPU index
wrappers, not in \texttt{run\_milvus\_hdf5.py}.

\section{Layer 3 --- Milvus from source (end-to-end)}

\subsection{Goal}

Milvus server uses the HIP-enabled Knowhere build on AMD GPU.

\subsection{Steps}

\begin{enumerate}[leftmargin=*]
  \item Build Milvus with GPU/HIP-enabled Knowhere (custom Dockerfile or bare-metal build).
  \item Run Standalone with GPU devices visible (example):
\begin{verbatim}
docker run --device=/dev/kfd --device=/dev/dri --group-add video ...
\end{verbatim}
  \item Create a collection with a GPU index, e.g.\ \texttt{GPU\_IVF\_FLAT} (closest to the
        CPU harness).
  \item Use Milvus GPU index params (\texttt{cache\_dataset\_on\_device}, etc.; see Milvus
        GPU index docs for your version).
\end{enumerate}

\textbf{Pass:} \texttt{create\_index}, \texttt{load}, and \texttt{search} succeed; logs show
GPU index path (not CPU SSE-only Knowhere).

\section{Engineering workflow}

Recommended flow:

\begin{enumerate}[leftmargin=*]
  \item Layer~0: ROCm + Milvus CPU healthy on \texttt{amd-rx7900xtx}.
  \item Layer~1: hipVS/hipRAFT standalone on \texttt{gfx1100} (Section~\ref{sec:layer1-measured}).
  \item Layer~2: Knowhere fork + HIP link; one index type.
  \item Layer~3: Milvus source build; Standalone GPU search smoke test.
  \item Layer~4: \texttt{ann-harness-amd} SIFT-1M \texttt{nprobe} sweep vs CPU Tables~5--6.
  \item If recall OK and GPU faster at target points: widen index types + VectorDBBench GPU
        cases.
  \item If not: debug numerics, params, device cache before advancing.
\end{enumerate}

\textbf{Branching:} maintain a \texttt{milvus-amd-gpu} fork (Knowhere submodule + CMake
patches). Pin versions: Milvus tag $\leftrightarrow$ Knowhere tag $\leftrightarrow$ hipVS
release $\leftrightarrow$ ROCm version in a single compatibility table per experiment.

\textbf{CI/local:} small dataset on every commit; full SIFT-1M nightly on the lab host in
\texttt{tmux}.

\section{Scope and realism}

\begin{itemize}[leftmargin=*]
  \item \textbf{hipVS is early} (e.g.\ v0.1.x)---expect gaps vs cuVS; start with
        \textbf{float} \texttt{GPU\_IVF\_FLAT} or GPU brute force, not every index at once.
  \item \textbf{RDNA3 gtest gap}---\texttt{NEIGHBORS\_ANN\_IVF\_FLAT\_TEST} packer asserts fail on
        \texttt{gfx1100} even at recall~1.0; use \texttt{BRUTEFORCE\_C\_TEST} + application recall
        for sign-off (Section~\ref{sec:ivf-gtest-decode}).
  \item \textbf{API compatibility $\neq$ drop-in binary}---Milvus/Knowhere still need
        build-system and wrapper changes.
  \item \textbf{CPU baseline stays essential}---GPU work asks whether the hipVS path improves
        the recall--QPS trade-off vs CPU Milvus/FAISS on the same host, not whether Milvus
        beats FAISS on raw in-process QPS.
  \item \textbf{Stock Docker Milvus} remains the CPU control; GPU porting is source-build +
        fork work, not a config toggle.
\end{itemize}
