        \item Knowhere (GPU indexes)
        \item hipVS + hipRAFT (AMD)

    \end{column}
    \begin{column}{0.48\textwidth}

      \begin{itemize}

        \item Host: RX 7900 XTX (\texttt{gfx1100})
        \item Line: Milvus \texttt{v2.5.4}
        \item Control: CPU Milvus (same host)
      \end{itemize}

\begin{frame}[fragile]{How vector search works (end to end)}
  \begin{columns}[T]
    \begin{column}{0.48\textwidth}
      \textbf{Procedure}
      \begin{enumerate}
        \item \textbf{Store} many vectors in the database
              (here: 1M image descriptors).
        \item \textbf{Query} arrives: one new vector
              (``find items like this'').
        \item Engine finds the \textbf{k nearest}
              stored vectors (we use $k=10$).
        \item Returns their IDs (and distances)
              to the application.
      \end{enumerate}
      \vspace{0.3em}
      {\small Exact scan of 1M vectors is slow;
      IVF + GPU approximates this fast.}
    \end{column}
    \begin{column}{0.50\textwidth}
      \centering
      \begin{tikzpicture}[
        font=\scriptsize,
        box/.style={draw, rounded corners=2pt, align=center,

\end{frame}

\begin{frame}[fragile]{What is recall@10?}
  \begin{columns}[T]
    \begin{column}{0.50\textwidth}
      \textbf{In one sentence}
      \begin{itemize}
        \item For each query we know the
              \emph{true} 10 nearest neighbors
              (from a trusted ground truth).
        \item The system also returns 10 IDs.
        \item \textbf{recall@10} = fraction of the
              true top-10 that appear in the
              system's top-10.
      \end{itemize}
      \vspace{0.35em}
      \textbf{Example:} true top-10 has 10 IDs;
      system finds 8 of them $\rightarrow$
      recall@10 $= 0.8$ (80\%).
      \vspace{0.35em}

      {\small $1.0$ = perfect; lower = more misses.\\
      Our GPU table goes about $0.38 \rightarrow 0.98$
      as search gets more thorough.}
    \end{column}
    \begin{column}{0.48\textwidth}
      \centering
      \begin{tikzpicture}[
        font=\tiny,
        hit/.style={circle, draw=green!60!black, fill=green!25,

\begin{frame}[fragile]{What is IVF?}
  \begin{columns}[T]
    \begin{column}{0.50\textwidth}
      \textbf{IVF = Inverted File} (Approximate Nearest Neighbor index)
      \begin{itemize}
        \item At \textbf{build} time, group the
              1M vectors into many clusters
              (\texttt{nlist}=1024 lists).
        \item Each list has a \emph{center}
              (centroid) and the vectors that
              belong to it.
        \item At \textbf{search} time, do \emph{not}
              compare the query to all 1M vectors.
        \item Compare to list centers first, then
              scan vectors only in a few lists
              (see \texttt{nprobe} next).
      \end{itemize}
      \vspace{0.3em}
      {\small Our indexes: CPU \texttt{IVF\_FLAT},
      GPU \texttt{GPU\_IVF\_FLAT} --- same idea;
      FLAT = store full vectors in each list.}
    \end{column}
    \begin{column}{0.48\textwidth}
      \centering
      \begin{tikzpicture}[

\begin{frame}[fragile]{What is nprobe? (IVF search knob)}
  \begin{columns}[T]
    \begin{column}{0.52\textwidth}
      \textbf{Given an IVF index}
      \begin{itemize}
        \item The query first finds the nearest
              list centers.
        \item \textbf{nprobe} = how many of those
              lists to open and scan
              (we used 1, 4, 8, 16, 32).
        \item Rest of the database is skipped
              --- that is the speedup vs a full scan.
      \end{itemize}
      \vspace{0.4em}
      \textbf{Trade-off}
      \begin{itemize}
        \item Higher \texttt{nprobe}: better recall, more work (usually lower QPS).
        \item Lower \texttt{nprobe}: faster, may miss true neighbors.
      \end{itemize}
    \end{column}
    \begin{column}{0.46\textwidth}
      \centering

\begin{frame}{Appendix --- index types beyond FLAT}
  \footnotesize
  This deck uses \textbf{IVF\_FLAT} / \textbf{GPU\_IVF\_FLAT}
  (full vectors in each list). Other common options:

  \vspace{0.4em}
  \begin{center}
  \begin{tabular}{@{}lll@{}}
    \toprule
    \textbf{Idea} & \textbf{Examples} & \textbf{Trade-off} \\
    \midrule
    Full vectors
      & \texttt{IVF\_FLAT}, \texttt{GPU\_IVF\_FLAT}
      & Best quality for a given scan; more memory \\
    Product quantization
      & \texttt{IVF\_PQ}, \texttt{GPU\_IVF\_PQ}
      & Much smaller / faster; slight recall loss \\
    Scalar quantization
      & SQ8 / similar
      & Smaller than FLAT; simpler than PQ \\
    Graph indexes
      & HNSW, CAGRA
      & Graph walk (not IVF lists) \\
    Binary / hash-style
      & binary IVF, LSH-style
      & Very compact; coarser distances \\
    \bottomrule
  \end{tabular}
  \end{center}

  \vspace{0.5em}
  {\small FLAT first keeps CPU vs GPU an apples-to-apples algorithm story;
  PQ / CAGRA are natural next steps.}
\end{frame}
